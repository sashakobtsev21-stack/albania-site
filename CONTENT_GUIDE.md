# CONTENT_GUIDE.md — как добавить SEO-статью на Albania Guidebook

> ⚠️ **Сайт переведён на ОДИН язык — английский (en) (2026-06-22).** ru/uk полностью удалены (страницы, контент, словари, LangSwitcher). Весь контент — только в `…/en/…`, отдаётся на корне `/<категория>/<slug>/`. Старые URL `/ru/*` и `/uk/*` → 301 на корень. Новый контент создаётся **только на английском**. Ниже по тексту встречаются устаревшие упоминания «пары/тройки ru/uk» — игнорируйте: актуально только en.

Рабочая инструкция для регулярного написания статей. **Источники правды по продукту/правилам:** `SPEC.md` → `CLAUDE.md` → этот файл. Если что-то противоречит — приоритет у SPEC/CLAUDE.

> TL;DR: статья = Markdown-файл в `src/content/articles/en/<slug>.md` (Astro Content Collections), отдаётся на корне `/<категория>/<slug>/`. Сайт одноязычный (en) — публикуется одна en-версия. Перед публикацией — `npm run qa` → должно быть **GO**.

> **📏 Стандарты публикации (обязательный минимум; авторитет — `CLAUDE.md` § «Стандарты публикации»):**
> **Объём** — статья **1200–2000 слов** (лид = прямой ответ за 2–3 предложения; H2/H3 под вопросы; без «воды» и без FAQ); новость **1500–2000 знаков** (~250–350 слов, **3–5 разделов**, **≥4 внутр. ссылки**).
> **Минимум фото** (каждое уникально на материал, **webp ≤200 КБ**, своё или CC/PD — Wikimedia/Openverse/Unsplash — с атрибуцией): статья **≥5** (cover + ≥4 инлайн/галерея); новость **≥2** (cover + ≥1 инлайн); маршрут — cover + фото **каждой** остановки; город **≥10**, **столица (Тирана) — ≥15**.
> **Модель «город = статья»** с `category: 'cities'` (коллекция `cities` для городов-путеводителей не используется).
> **Тон** — живой, как у носителя (uk — живая мова, не калька/суржик; en — как пишет носитель), **НЕ ИИ-генерёнка**; факты только из источника (+ «уточняйте» + дата); валюта — **албанский лек (ALL)**.
> ⚠️ Где ниже в инструкции встречается «2–4 фото» — это старая формулировка; действует норма **≥5 фото на статью**.

## 0. Для владельца: как это работает (простыми словами)

**Главное: статьи пишешь НЕ ты.** Их пишу я: ведущий черновик en → зеркала ru/uk → фото → проверка фактов → проверка качества → публикация. Твоя роль короткая:

1. **Дать факты, которых нет в открытых источниках** — особенно местное по Дурресу/Ривьере (любимые места, цены по чеку, что реально стоит/не стоит, что закрылось). Без этого я пишу только на **проверяемых фактах** из надёжных источников — без выдуманного «я там был».
2. **Дать свои фото** (по желанию) — они дают метку «Проверено» и сильнее в Google, чем сток. Нет своих → беру бесплатные CC.
3. **Сказать «ок»** на публикацию.

**Как заказать статью:** просто напиши «пиши про &lt;тему&gt;» — я беру тему из плана, собираю факты, пишу группу en + ru + uk (en — ведущая, от EN-интента; ru/uk — зеркала), прогоняю гейты и публикую. Или кинь мне факт/место — я превращу в контент.

**По какому плану:** очередь и приоритеты — `docs/CONTENT_PLAN.md` (§0 — стратегия, дальше кластеры/очередь). Порядок исполнения — `ROADMAP-GROWTH-2026.md`. Куда движемся — память `growth-strategy-direction`.

**Что пишем сейчас (ближайшее):**
1. Гайды еды (Тирана/Дуррес/Саранда) — где поесть, содержательно по местам.
2. Гайд «вино Албании» — мост в винные/гастро-туры (деньги).
3. Цена/часы (`visit`) + фактические заметки у топ-достопримечательностей и страниц побережья (Дуррес/Ривьера) — против AI Overviews.
4. Английский ведущий поток (EN-first) — темы от EN-интента, ru/uk зеркалами.

**Правило фактов (железное, §15):** цены/адреса/часы — только из надёжного источника + «уточняйте» + дата; нет данных → не выдумываем. Это защита от бана Google.

## 1. Как технически устроены статьи

- **Движок:** Astro 6 + **Content Collections**. Статьи — это **Markdown** (`.md`/`.mdx`) с YAML-frontmatter. Отдельные `.astro`-файлы под каждую статью НЕ создаются — рендерит общий шаблон `src/layouts/ArticlePage.astro` через динамический маршрут.
- **Где лежат файлы:**
  ```
  src/content/articles/ru/<slug>.md   ← русская версия
  src/content/articles/uk/<slug>.md   ← украинская (обязательна, см. §3)
  src/content/articles/en/<slug>.md   ← английская (по желанию)
  ```
  (Маршруты — отдельная коллекция `src/content/routes/`; «Где поесть» — `src/content/restaurants/`; услуги — `src/content/services/`; города-карточки — `src/content/cities/`. Эта инструкция — про **статьи** в `articles/`.)
- **Как файл попадает в нужный раздел:** разделом управляет поле **`category`** во frontmatter, а НЕ папка. URL строится как `/<lang-префикс>/<category>/<slug>/`. Файл `articles/ru/abc.md` с `category: 'cities'` → страница `/ru/cities/abc/` и автоматически попадает в хаб `/ru/cities/` (хаб сам подтягивает статьи своей категории — ручных списков нет).
- **Допустимые `category`** (enum в `src/content.config.ts`):
  `attractions` · `cities` · `food` · `entertainment` · `routes` · `transport` · `car-rental` · `relocation` · `insurance` · `news`.
  - Для обычных SEO-статей чаще всего: **cities, attractions, transport, car-rental, relocation, news, entertainment**.
  - Нюансы: `routes` — это отдельная коллекция `routes/` (не `articles/`); `food` — директория мест (`restaurants/`), а не лонгрид; `insurance` — раздел с кастомным хабом, но статьи категории `insurance` в `articles/` тоже работают (`/ru/insurance/<slug>/`).

## 2. Frontmatter статьи (поля)

Полная схема — `src/content.config.ts` (`articleBase`). Главное:

| Поле | Обяз.? | Что это |
|---|---|---|
| `title` | да | Заголовок = **H1** на странице. **≤ 60 символов** (гейт `check-parity`). Должен содержать целевой запрос (см. §8). |
| `description` | да | Meta description. **≤ 155 символов** (валидируется схемой). 1–2 предложения с запросом. |
| `slug` | да | Латиница-цифры-дефис. **Стабилен навсегда** (URL не меняем — правило 3). Одинаковый во всех языках. |
| `lang` | да | `'ru'` / `'uk'` / `'en'` — должен совпадать с папкой. |
| `category` | да | Один из enum выше → определяет раздел и URL. |
| `publishedAt` | да | Дата публикации `YYYY-MM-DD`. |
| `updatedAt` | да | Дата обновления `YYYY-MM-DD` (видна на странице — «Обновлено · …»). Поднимай при правках фактов. |
| `cover` | да* | Обложка: `src` (путь к webp) + `alt`. *Фото обязательны (правило проекта); гейт `check-photos` валит сборку без существующего cover ≤200 КБ. |
| `coverCredit` | для CC | Атрибуция обложки «Фото: Автор / источник, лицензия» — обязательна для CC/чужих фото (§18). |
| `gallery` | нет | Массив `{src, alt, credit?}` — мини-галерея с лайтбоксом. |
| `affiliate` | нет | 1–3 партнёрских бокса (см. §7). По умолчанию `[]`. |
| `sources` | нет | Массив доменов/ссылок-первоисточников фактов (§14). |
| `verifiedAt` | нет | Дата реальной сверки места → бейдж «Проверено · …». Без сверки — НЕ ставить (будет «Обновлено», правило 4). |
| `geo` | нет | `{ coord: [lat, lng], address? }` — для статей о местах (даёт карту + schema). |
| `accessFrom` | нет | «Как доехать» из Тираны/Дурреса/Саранды (`{km?, duration?, note?}`). |
| `visit` | нет | «Вход и часы»: `{price?, hours?, note?, checkedAt?}`. Цены/часы НЕ выдумывать — только источник + дата + «уточняйте». |
| `featuredOrder` | нет | Меньше число = выше в хабе раздела. |
| `attractionType` / `region` | нет | Только для `attractions` (фильтры каталога). Слаги — в `content.config.ts`. |
| `razvlType` | нет | Только для `entertainment` (`kazino`/`mesta`/`nochnaya-zhizn`/`afisha`/`aktivnyy`). |
| `tags` | нет | Свободные теги (показ в смежных разделах). |
| `hotelWidget` | нет | `true` → виджет отелей Trip.com (для городов с высоким спросом на жильё). |
| `draft` | да на старте | `true` = скелет, в сборку НЕ попадает. Снять (`false`) только когда статья готова и гейты зелёные. |

**Примеры (реальные статьи — посмотри их целиком):**
- `src/content/articles/ru/kak-vybrat-strahovku.md` — справочная статья: cover + `affiliate` (SafetyWing) + `sources` + внутренние ссылки.
- `src/content/articles/ru/valbona.md` — статья о месте: `geo`, `gallery`, инлайн-фигуры, `accessFrom`.
- `src/content/articles/ru/tirana.md` — город: `featuredOrder`, `hotelWidget`, `visit`.

## 3. Мультиязычность (EN/RU/UK) и hreflang

- **EN — язык по умолчанию, на корне `/`.** RU → `/ru/…`, UK → `/uk/…`. (Префикс задаёт `langPrefix()` в `src/i18n/index.ts`: `DEFAULT_LANG='en'`. NB: пара комментариев в коде ещё говорит «ru — корень» — это устаревшая надпись, рантайм отдаёт en на корне.)
- **Чтобы EN-статья попала на корень:** файл в `src/content/articles/en/<slug>.md`, `lang: 'en'`, `category: '<…>'`. URL получится `/<category>/<slug>/` без префикса. RU/UK-зеркала — `src/content/articles/{ru,uk}/<slug>.md` с тем же slug → `/ru/<category>/<slug>/` и `/uk/<category>/<slug>/` автоматически (маршруты `src/pages/ru/[category]/[slug].astro` и т.п.).
- **🔴 Главное правило (CLAUDE правило 1, EN-first):** статья — это группа **en + ru + uk** с **одинаковым `slug`**; **en — ведущая** (пишется первой, от EN-интента), ru/uk — зеркала. Публикуется группа целиком. Неполная группа **не пройдёт** гейт `check-parity` (он требует консистентную группу). Половину группы не публиковать.
- **hreflang — автоматический.** `BaseLayout` сам проставляет взаимные `<link rel="alternate" hreflang="…">` + `x-default → en` по тем языкам, где **существует тот же `slug`**. От тебя нужно только: одинаковый slug в en/ru/uk — и связка en↔ru↔uk (x-default → en) построится сама. Переключатель языка тоже работает по совпадающему slug (`mirrorPath`).
- **Перевод UK:** адаптируй под украинскую аудиторию (а не дословно): EKTA вместо недоступных страховок, валютные нюансы. Числовые поля (даты, цифры) и `slug`, `cover.src`, `gallery` src — одинаковые во всей группе.

## Как делать UK-версию правильно

UK-версия — обязательное зеркало в группе (правило 1) и **полноценная статья на украинском**, а не технический довесок:

1. **Живой украинский — не калька и не суржик.** Текст должен читаться как написанный носителем украинского: естественные конструкции, украинская лексика и идиомы. НЕ пословный перевод и НЕ смесь языков — переводим смысл и тон, а не слова.
2. **Факты идентичны ведущей EN.** Цены, цифры, расстояния, даты, названия — те же, что в ведущей EN-версии (как и `slug`, `cover.src`, `gallery` src, числовые поля frontmatter). Расхождение фактов между версиями недопустимо.
3. **H1 = украинский целевой запрос.** `title` украинской версии содержит то, **как ищут носители украинского в Google** (украинский поисковый запрос), а НЕ перевод английского H1. При этом `title` **≤ 60 символов**: если запрос длинный — бери его **ядро в H1**, а длинный хвост раскрывай в подзаголовках H2/H3. Аналогично — украинская `description` (≤155) под украинский запрос.
4. **Внутренние ссылки — на UK-страницы.** В UK-версии все внутренние ссылки с префиксом `/uk/…` (а не `/ru/…` или корня). Минимум 2, как и в остальных версиях.
5. **Тело — украинские H2/H3** под смежные украинские запросы, своя лид-фраза, подписи к фото на украинском. Кредиты фото после «Фото:» (автор/лицензия) НЕ переводим — оставляем дословно.

**RU-версия — то же зеркало в группе** (обязательна, правило 1): те же правила, что для UK, но под русский поисковый запрос; внутренние ссылки с префиксом `/ru/…`; факты идентичны ведущей EN. **EN — ведущая версия группы** (пишется первой, от EN-интента): `title` ≤60 под английский запрос; внутренние ссылки на корне (`/…`, без префикса); ru/uk зеркалят её факты.

## 4. Пошагово: добавить статью (EN-first)

1. **Бриф/факты.** Собери факты заранее (цены, адреса, часы — из надёжных источников). Нет факта → ставь `<!-- TODO -->` и не выдумывай (правило 4). Можно завести `briefs/<slug>.md`.
2. **Скелет (рекомендуется).** Из корня проекта:
   ```
   npm run new -- article <slug> --category <category> --title "Заголовок ≤60 символов"
   ```
   Создаст `articles/{en,ru,uk}/<slug>.md` (схема-корректный frontmatter, `draft: true`, DoD-чеклист в комментарии) + папку фото `public/images/<slug>/`. Существующие файлы не перезатирает.
   *(Альтернатива — скопировать существующую статью и поменять frontmatter.)*
3. **Текст EN (ведущий).** Заполни `articles/en/<slug>.md` от EN-интента:
   - `description` (≤155), `publishedAt`/`updatedAt`.
   - **Тело:** начинается с лид-абзаца (это первый абзац, без `#`), затем секции `## H2` / `### H3`. **`#`/H1 в теле НЕ ставить** — H1 рендерится из `title`.
   - Соблюдай SEO-требования из §8.
4. **Фото** (§6) — cover обязателен + 2–4 инлайн-фигуры или галерея.
5. **Внутренние ссылки** (§7) — минимум 2 на смежные статьи + ссылка на хаб раздела.
6. **Зеркала RU и UK** — создай/заполни `articles/ru/<slug>.md` и `articles/uk/<slug>.md` (тот же slug), зеркаля факты ведущей EN.
7. **Сними `draft`** → `draft: false` во всех трёх версиях.
8. **Гейты** (§9) — `npm run qa` → **GO**.
9. **Коммит группы** en+ru+uk одним коммитом → push в `main` (Cloudflare соберёт и задеплоит).

## 6. Фотографии

- **Уникальные на статью (правило владельца 2026-06-19):** у каждой статьи СВОИ фото — **НЕ переиспользовать кадры из других статей** (`/images/<другой-slug>...`). Под новую статью искать новые кадры через пайплайн и класть в свою папку. Проверять, что НА фото (не выдавать чужую страну/валюту за албанскую). Память `article-photos-unique`.
- **Где лежат:** `public/images/<slug>/` (создаётся скаффолдером). Формат — **`.webp`**, каждое **≤ 200 КБ** (перф-бюджет §15), с заданными `width`/`height` (нулевой CLS).
- **Откуда брать:** только свои фото или **CC/PD** (Wikimedia / Openverse / Unsplash). Скрипты-помощники: `scripts/commons-candidates.mjs`, `scripts/openverse-candidates.mjs`, `scripts/unsplash-candidates.mjs` (поиск кандидатов) → `scripts/build-gallery.mjs` / `build-unsplash.mjs` (скачать + сжать в webp ≤200 КБ). Атрибуцию (автор+лицензия) бери **дословно из источника**, не из имени файла.
- **Обложка (frontmatter):**
  ```yaml
  cover:
    src: '/images/<slug>/cover.webp'
    alt: 'Осмысленное описание кадра'
  coverCredit: 'Фото: Имя Автора / Wikimedia Commons, CC BY-SA 4.0'
  ```
- **Инлайн-фото в тексте** (ключевые кадры — прямо в теле, с подписью). Точная разметка (raw HTML в .md):
  ```html
  <figure class="figure">
  <img src="/images/<slug>/g1.webp" alt="Что на фото" width="1280" height="853" loading="lazy" decoding="async" />
  <figcaption>Подпись — что на фото. <span class="figure__credit">Фото: Автор / источник, лицензия</span></figcaption>
  </figure>
  ```
  `width`/`height` = реальные пиксели файла (иначе перекос/CLS).
- **Мини-галерея внизу** (необязательные кадры) — через frontmatter:
  ```yaml
  gallery:
    - src: '/images/<slug>/g2.webp'
      alt: '…'
      credit: 'Фото: …'
  ```
  Каждое фото используется **один раз** (либо инлайн, либо в галерее).
- **`alt`** — осмысленный, по делу (не «фото1»); описывает содержимое кадра.
- **После добавления обложек** прогони `npm run build:covers` — сгенерит адаптивные варианты (`srcset`) и обновит манифест `src/data/cover-variants.json` (мобайл грузит меньший размер). Варианты коммитятся.

## 7. Внутренние ссылки

- **Формат (RU):** обычная Markdown-ссылка с **префиксом языка** и завершающим слэшем:
  ```markdown
  Подробнее — в путеводителе по [Тиране](/ru/cities/tirana/) и в гайде по [аренде авто](/ru/car-rental/).
  ```
  Для UK-версии — `/uk/…`, для EN — без префикса (`/cities/tirana/`).
- **Минимум 2** внутренние ссылки в теле на смежные статьи — это **гейт** `check-interlinks` (DoD §8.1). Партнёрские `/go/`-ссылки и якоря в счёт не идут.
- Ставь ссылку и **на хаб раздела** (`/<category>/` на EN-корне; `/ru/<category>/`, `/uk/<category>/` в зеркалах), и желательно встречную ссылку из смежной статьи (двунаправленный кластер).
- **Партнёрские ссылки — ТОЛЬКО через `/go/`** (правило 2), и только во frontmatter `affiliate` (рендерится как AffiliateBox), не прямыми URL в тексте:
  ```yaml
  affiliate:
    - partner: 'trip-hotels'      # ключ из src/data/partners.json
      label: 'Где остановиться'
      position: 'body-end'
  ```

## 8. SEO-требования к статье (соблюдаем ВСЕГДА)

1. **H1 = целевой запрос.** Поле `title` (= H1) точно содержит целевой поисковый запрос. Ограничение: **≤ 60 символов** (гейт). Один H1 на странице (он из `title`; в теле `#` не использовать).
2. **Первый абзац — прямой ответ на запрос в 2–3 предложениях.** Лид сразу даёт суть, без «разгона».
3. **Подзаголовки `## H2` / `### H3` со смежными запросами** — структура под вопросы читателя (что/где/как/сколько/когда). Логичная иерархия без скачков уровней.
4. **Таблицы** для цен / расстояний / сравнений — обычные Markdown-таблицы (`| … | … |`). Числа — только из источника (правило 4), цены — с «уточняйте» + дата.
5. **2–4 фото** с осмысленными `alt` и подписями (figcaption) — см. §6.
6. **Блок внутренних ссылок** на смежные статьи — минимум 2 (см. §7), плюс хаб раздела.
7. **Дата обновления** — поле `updatedAt` (видно на странице). Поднимать при правке фактов.
8. **Объём 1200–2000 слов, без «воды».** Каждый абзац несёт пользу; никакой генерёнки и near-duplicate (§15 — риск бана).
9. **Доп. требования проекта:** `description` ≤155 с запросом; факты не выдумывать (правило 4) — нет данных → TODO/«уточняйте» + источник в `sources`; уникальная фактура (≥3 своих факта), не перепечатка; **FAQ не используем** (убран со всего сайта — новую инфо вплетать в тело).

## 9. Проверка перед публикацией (гейты)

Из корня проекта:

| Команда | Что проверяет |
|---|---|
| `npm run check` | типы + соответствие frontmatter схеме |
| `npm test` | enums · **parity (полная группа en+ru+uk, одинаковый slug, title ≤60)** · photos (cover ≤200 КБ) · **interlinks (≥2)** |
| `npm run test:links` | внутренние ссылки резолвятся + `/go/`-партнёры валидны |
| `npm run lint` | стиль/ESLint |
| **`npm run qa`** | **всё вышеперечисленное + чистая сборка + аудит `dist/` → вердикт GO/NO-GO** |

**Правило:** перед коммитом — `npm run qa` = **GO**. Затем коммит группы en+ru+uk + push в `main`.

**Как убедиться, что статья на месте с корректным hreflang:**
1. `npm run build` → проверь, что появились `dist/<category>/<slug>/index.html` (EN-корень), `dist/ru/<category>/<slug>/index.html` и `dist/uk/<category>/<slug>/index.html`.
2. В каждом есть взаимные `<link rel="alternate" hreflang="en" …>`, `hreflang="ru" …>`, `hreflang="uk" …>` + `x-default` (→ en) — проставляется автоматически по совпадающему slug.
3. `npm run test:links` — без битых ссылок.

## 12. Как добавлять ЗАВЕДЕНИЕ (ресторан / услугу) — это делаю я

Заведения — это НЕ лонгрид-статьи, а карточки-директории:
- **Ресторан / кафе / бар / винотека** → коллекция `restaurants/` → раздел «Где поесть» `/food/`.
- **Услуга** (риелтор / клининг / переезд / ремонт / документы / переводчики) → коллекция `services/` → `/relocation/services/`.

### Жёсткие правила отбора (обязательно)
1. **Только реальные заведения** с проверяемыми данными. Вымышленные/«примеры» в директорию проверенных мест НЕ вносим (правило 4, §15).
2. **Консенсус > 4★ из НЕСКОЛЬКИХ источников** + наш `ratingNote`/`review` своими словами (чужие отзывы дословно нельзя, §15).
3. **Фото обязательно** (`cover`; для CC — `coverCredit`). Цены/часы (`dishes`/`hours`) — только из источника, иначе не указывать.
4. **Группа en + ru + uk** (en — ведущая), одинаковый `slug`.
5. **🔵 Метка «Проверено» — ОБЯЗАТЕЛЬНА** (правило владельца, §13 ниже).

### Шаги
1. Скаффолд: `npm run new -- restaurant <slug> --city "Тирана"` (или `service <slug>`) → создаст ru/uk/en + папку фото.
2. Заполни обязательные поля. **Ресторан:** `name`, `city`, `priceLevel`, `geo.coord`, `mapUrl`, `ratingNote`, `sources`, `summary`, **`verifiedAt`**, `cover`. **Услуга:** `name`, `rubric`, `summary`, реальный `url`/`contact` (без example.com), **`verifiedAt`**.
3. Фото (§6); `verifiedAt` = дата нашей **реальной** сверки данных.
4. Перевод uk (+en) → снять `draft` → `npm run qa` = GO → коммит пары.

### Добавить заведение в витрину на главной (с флагом города)
В `HomePage.astro` → `showcasePicks` добавить pick с полем `city` (`city: 'Тирана'`) → на карточке появится пин-чип «📍 Город» (см. память `home-showcase-monetization`). В витрину помещаем только заведения с меткой «Проверено» (§13).

## 13. 🔵 Правило: метка «Проверено» обязательна у КАЖДОГО заведения/услуги

**КАЖДОЕ заведение и услуга, которые мы вносим, обязаны нести видимую метку «Проверено» (галочка ✓)** — это обещание директории «проверенные места» и ключевой сигнал доверия (E-E-A-T).

- **Чем:** единый компонент **`VerifiedBadge`** (вино-галочка ✓ + «Проверено · {дата}» + расшифровка «опытными путешественниками, гидами и местными»). Дата — из поля `verifiedAt`. **Не изобретать новых галочек** — один знак доверия на весь сайт (консистентность).
- **Честность (правило 4):** `verifiedAt` ставим ТОЛЬКО при реальной сверке (данные сверены с источниками/на месте). «Проверено» на непроверенном — запрещено.
- **Текущее состояние:** рестораны (`/food/`) — `verifiedAt` обязателен схемой → полный бейдж «Проверено · {дата}» ✓. Услуги (`/services/`) — ✅ **добавлено (2026-06-18):** поле `verifiedAt` в схеме `services` + **компактный** `VerifiedBadge` (проп `compact`: ✓ + «Проверено», без даты) в карточках; ставить только реально сверенным услугам (демо/вымышленные — без). Витрина: компактная ✓ на карточке заведения — когда заведения туда попадут.
- **Компонент:** `VerifiedBadge` теперь имеет `compact` (для плотных карточек — ✓ + «Проверено») и полный (✓ + «Проверено · {дата}» + расшифровка — для страницы места/шапки статьи).
- **Метка «Проверено» — БЕЗ даты у ВСЕХ заведений** (решение владельца 2026-06-18; даты не добавлять никогда): `RestaurantCard` всегда рендерит компактный бейдж «Проверено» без даты. `verifiedAt` остаётся обязательным в схеме (хранится для записи), но в карточке НЕ показывается.
